# demo_data

Pre-computed sample data for the **TransitLens** platform.

These files are used by the Demo page and as offline fallbacks when the
`ml-core` backend is unavailable.  They were originally generated by the
**data-pipeline** module and committed here so the Streamlit front-end can
run in fully offline / standalone mode.

---

## Files

### `candidate_a.csv`

Synthetic TESS-format light curve simulating a **Jupiter-sized exoplanet**
transit.  Contains `time` (BTJD) and `flux` (normalised) columns with ~500
data points.  The signal has a shallow 1.3 % periodic dip every ~3.42 days.

### `candidate_b.csv`

Synthetic light curve simulating an **eclipsing stellar binary**.  Same
two-column format as above.  Features a deep ~18 % V-shaped eclipse with a
period of ~1.87 days — characteristic of two stars in a close orbit.

### `candidate_c.csv`

Synthetic light curve representing **noise / stellar variability** with no
genuine periodic transit signal.  Correlated red noise mimics instrumental
systematics and intrinsic stellar variability.

### `sample_results.json`

Pre-computed analysis results for all three candidates (keys:
`candidate_a`, `candidate_b`, `candidate_c`).  Each entry mirrors the JSON
response returned by `ml-core POST /analyze` and includes:

| Field               | Description                                     |
|---------------------|-------------------------------------------------|
| `target_id`         | Candidate identifier (`candidate_a`, etc.)      |
| `candidate_detected`| Boolean — was a periodic transit signal found?  |
| `predicted_class`   | `exoplanet_like`, `eclipsing_binary_like`, or `noise_or_other` |
| `confidence`        | Model confidence score (0 – 1)                  |
| `period_days`       | Best-fit orbital period in days                  |
| `duration_days`     | Transit duration in days                         |
| `depth`             | Fractional transit depth                         |
| `snr`               | Signal-to-noise ratio of the detection           |
| `features`          | Dict of 11 extracted features (BLS power, etc.)  |
| `plots`             | Dict of 4 base64-encoded PNG diagnostic plots    |
| `explanation`       | Natural-language summary of the classification   |

## How to Regenerate `sample_results.json`

1. **Start ml-core** on `localhost:8000`:

   ```bash
   cd ../ml-core
   uvicorn main:app --host 0.0.0.0 --port 8000
   ```

2. **Run the regeneration script** from the project root:

   ```bash
   python -c "
   import json, numpy as np
   from app.utils import load_csv
   from app.api_client import analyze

   results = {}
   for cid in ('candidate_a', 'candidate_b', 'candidate_c'):
       time_arr, flux_arr = load_csv(f'demo_data/{cid}.csv')
       result = analyze(time_arr, flux_arr, target_id=cid)
       results[cid] = result

   with open('demo_data/sample_results.json', 'w') as f:
       json.dump(results, f, indent=2)

   print('✅  sample_results.json regenerated')
   "
   ```

   > **Note:** The script sends each CSV to the live `ml-core` `/analyze`
   > endpoint and writes the full responses (including base64 plot data)
   > into the JSON file.  Make sure ml-core is running and healthy before
   > executing.

3. Commit the updated file:

   ```bash
   git add demo_data/sample_results.json
   git commit -m "chore: regenerate sample results"
   ```
